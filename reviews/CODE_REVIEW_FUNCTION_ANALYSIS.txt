Fruits Local LLM RAG MVP - 함수 단위 코드 분석
작성 목적: 사용자가 직접 코드 리뷰를 할 수 있도록 각 파일의 역할, 함수 흐름, 입력/출력, 리뷰 포인트를 정리한다.

================================================================================
1. app/main.py
================================================================================

파일 역할
- FastAPI 애플리케이션의 진입점이다.
- 앱 메타데이터를 설정하고 chat 라우터를 등록한다.
- 서버 상태 확인용 /health 엔드포인트를 제공한다.

전역 객체: app
- FastAPI 인스턴스다.
- title, description, version을 설정한다.
- app.include_router(chat_router)로 app/api/chat.py의 /chat API를 연결한다.

함수: health() -> dict[str, str]
- 경로: GET /health
- 역할:
  - FastAPI 서버가 살아 있는지 확인한다.
- 입력:
  - 없음.
- 출력:
  - {"status": "ok"}
- 리뷰 포인트:
  - DB, Ollama, MariaDB Vector 연결 상태까지 확인하지는 않는다.
  - 현재는 서버 프로세스 생존 확인용으로 충분하다.
  - 나중에 운영 점검용으로 쓰려면 /health/db, /health/ollama 같은 상세 체크를 추가하는 것이 좋다.

================================================================================
2. app/api/chat.py
================================================================================

파일 역할
- POST /chat 엔드포인트를 정의한다.
- 사용자 질문을 받아 RAG 검색, 프롬프트 생성, LLM 호출, 대화 저장, 요약 갱신을 한 요청 안에서 처리한다.

클래스: ChatRequest(BaseModel)
- 역할:
  - /chat 요청 JSON 구조를 정의한다.
- 필드:
  - question: 사용자 질문. 최소 길이 1.
  - session_id: 기존 대화를 이어갈 때 사용하는 MariaDB chat_sessions.id.
  - user_id: 사용자 식별자. 기본값은 local_user.
- 리뷰 포인트:
  - 현재 user_id는 인증과 연결되어 있지 않다.
  - session_id만 알면 기존 세션을 가져올 수 있으므로 다중 사용자로 확장할 때는 검증이 필요하다.

클래스: SourceChunk(BaseModel)
- 역할:
  - RAG 검색 결과 chunk의 응답 구조를 정의한다.
- 필드:
  - source_path: 원본 Markdown 파일 경로.
  - chunk_index: 문서 내 chunk 번호.
  - content: chunk 본문.
  - distance: 질문 embedding과 chunk embedding 사이의 cosine distance.
- 리뷰 포인트:
  - content 전체를 UI에 노출한다. 내부 문서를 숨겨야 하는 환경이면 응답에서 제외하거나 관리자 전용으로 분리해야 한다.

클래스: ChatResponse(BaseModel)
- 역할:
  - /chat 응답 JSON 구조를 정의한다.
- 필드:
  - session_id: 현재 대화 세션 ID.
  - answer: LLM 답변.
  - sources: 검색된 문서 chunk 목록.

함수: chat(request: ChatRequest) -> ChatResponse
- 경로:
  - POST /chat
- 전체 흐름:
  1. session_id가 있으면 기존 세션을 가져오고, 없으면 새 세션을 만든다.
  2. MariaDB에서 최근 3턴 대화와 요약본을 가져온다.
  3. 사용자 질문을 embedding으로 변환하고 MariaDB Vector 검색을 수행한다.
  4. 검색된 chunk와 질문으로 RAG prompt를 만든다.
  5. Ollama Qwen 모델에 prompt, 최근 대화, 요약을 전달해 답변을 생성한다.
  6. 사용자 질문과 assistant 답변을 chat_messages에 저장한다.
  7. 오래된 대화가 많으면 Qwen으로 요약해 chat_sessions.summary에 저장한다.
  8. session_id, answer, sources를 반환한다.
- 입력:
  - ChatRequest
- 출력:
  - ChatResponse
- 예외 처리:
  - 모든 예외를 잡아 HTTP 500으로 반환한다.
- 리뷰 포인트:
  - 요약 생성 실패도 전체 채팅 실패로 이어질 수 있다.
  - retrieve_context, generate_answer, add_message, maybe_summarize_session 중 어느 단계에서 실패했는지 사용자에게 구분해서 알려주지 않는다.
  - 모든 예외 detail을 그대로 노출하므로 DB 접속 정보나 내부 오류가 사용자에게 보일 수 있다.
  - 동기 함수 안에서 embedding, LLM, DB 작업을 모두 처리하므로 요청 시간이 길어질 수 있다.

3. app/db/connection.py
================================================================================

파일 역할
- MariaDB 연결을 생성하고 cursor context manager를 제공한다.
- 모든 DB 접근 코드가 이 파일을 통해 연결된다.

함수: get_connection() -> mariadb.Connection
- 역할:
  - .env 또는 기본값으로 MariaDB 연결을 생성한다.
- 사용하는 환경변수:
  - MARIADB_HOST
  - MARIADB_PORT
  - MARIADB_USER
  - MARIADB_PASSWORD
  - MARIADB_DATABASE
- 출력:
  - mariadb.Connection 객체.
- 리뷰 포인트:
  - 요청마다 새 DB 연결을 만든다.
  - MVP에서는 단순해서 좋지만, 사용량이 늘면 connection pool을 고려해야 한다.
  - 접속 실패 시 예외가 그대로 상위로 올라간다.

함수: db_cursor() -> Iterator[mariadb.Cursor]
- 역할:
  - dictionary=True cursor를 열고, 작업 성공 시 commit, 실패 시 rollback 후 연결을 닫는다.
- 사용 방식:
  - with db_cursor() as cursor:
        cursor.execute(...)
- 출력:
  - context manager 형태의 cursor.
- 리뷰 포인트:
  - SELECT만 하는 경우에도 commit이 실행된다. 큰 문제는 아니지만 엄밀히는 read-only 작업과 write 작업을 분리할 수 있다.
  - 여러 DB 작업을 하나의 트랜잭션으로 묶고 싶을 때 현재 구조는 함수 단위로 연결이 나뉘어 제한이 있다.

4. app/db/schema.sql
================================================================================

파일 역할
- MariaDB 테이블 구조를 정의한다.
- RAG 문서 저장과 대화 메모리 저장을 담당한다.

테이블: rag_documents
- 역할:
  - Markdown 문서 chunk와 embedding을 저장한다.
- 주요 컬럼:
  - source_path: 원본 문서 경로.
  - chunk_index: 문서 내 chunk 번호.
  - content: chunk 본문.
  - embedding: MariaDB Vector 데이터. 현재 VECTOR(1024).
- 인덱스:
  - UNIQUE KEY uq_source_chunk(source_path, chunk_index)
  - VECTOR INDEX(embedding) M=8 DISTANCE=cosine
- 리뷰 포인트:
  - VECTOR(1024)는 bge-m3 embedding 차원에 맞춘 값이다.
  - nomic-embed-text처럼 차원이 다른 모델로 바꾸면 스키마도 바꿔야 한다.
  - 문서 삭제/축소 시 오래된 chunk가 남는 문제는 ingest 로직에서 삭제 처리하도록 개선되었다.

테이블: chat_sessions
- 역할:
  - 대화 세션 단위의 메타정보와 요약을 저장한다.
- 주요 컬럼:
  - id: 세션 ID.
  - user_id: 사용자 ID.
  - summary: 오래된 대화 요약.
  - message_count: 누적 메시지 수.
- 리뷰 포인트:
  - user_id가 있지만 현재 인증/검증에는 사용되지 않는다.
  - summary가 TEXT라 아주 긴 요약 누적에는 제한이 있다. 현재 1,200자 제한 프롬프트가 있어 MVP에서는 충분하다.

테이블: chat_messages
- 역할:
  - 사용자와 assistant의 원문 메시지를 저장한다.
- 주요 컬럼:
  - session_id: chat_sessions.id 참조.
  - role: user 또는 assistant.
  - content: 메시지 본문.
  - is_summarized: 요약에 반영되어 prompt 원문에서는 제외할지 여부.
- 리뷰 포인트:
  - 오래된 메시지를 삭제하지 않고 is_summarized만 TRUE로 바꾼다.
  - 개인정보나 민감정보가 들어갈 수 있으면 보관 기간 정책이 필요하다.

5. app/db/vector_search.py
================================================================================

파일 역할
- MariaDB Vector에 chunk embedding을 저장하고 유사 chunk를 검색한다.

함수: vector_literal(embedding: list[float]) -> str
- 역할:
  - Python list 형태의 embedding을 MariaDB VEC_FromText가 받을 수 있는 JSON 문자열로 변환한다.
- 입력:
  - embedding: float 리스트.
- 출력:
  - 공백 없는 JSON 배열 문자열.
- 리뷰 포인트:
  - embedding 길이 검증이 없다.
  - VECTOR(1024)와 다른 길이가 들어오면 DB 단계에서 실패한다.

함수: upsert_document_chunk(source_path, chunk_index, content, embedding) -> None
- 역할:
  - rag_documents에 chunk를 삽입하거나 같은 source_path/chunk_index가 있으면 갱신한다.
- 입력:
  - source_path: 문서 경로.
  - chunk_index: chunk 번호.
  - content: chunk 본문.
  - embedding: chunk embedding.
- 출력:
  - 없음.
- SQL 특징:
  - INSERT ... ON DUPLICATE KEY UPDATE 사용.
  - embedding은 VEC_FromText(?)로 저장.
- 리뷰 포인트:
  - 같은 chunk 번호는 갱신된다.
  - 현재 ingest_markdown_file()이 저장 전에 해당 source_path의 기존 chunk를 삭제하므로, 문서 chunk 수가 줄어든 경우에도 오래된 chunk가 남지 않는다.
  - embedding 차원 검증을 Python에서 먼저 하면 오류를 더 친절하게 처리할 수 있다.

함수: delete_document_chunks(source_path: str) -> None
- 역할:
  - 특정 문서 source_path에 해당하는 기존 chunk를 모두 삭제한다.
- 입력:
  - source_path: 삭제할 문서 경로.
- 출력:
  - 없음.
- 사용 위치:
  - app/rag/ingest_docs.py의 ingest_markdown_file()에서 새 chunk를 저장하기 전에 호출한다.
- 리뷰 포인트:
  - 문서 하나를 재색인할 때 오래된 chunk를 확실히 제거한다.
  - 삭제 후 embedding 생성 중간에 실패하면 해당 문서 chunk가 일부만 다시 저장될 수 있다.

함수: delete_chunks_except_sources(source_paths: list[str]) -> int
- 역할:
  - 현재 rag_docs 폴더에 존재하지 않는 source_path의 chunk를 DB에서 삭제한다.
- 입력:
  - source_paths: 현재 존재하는 Markdown 문서 경로 목록.
- 출력:
  - 삭제된 row 수.
- 사용 위치:
  - ingest_all() 시작 시 호출한다.
- 리뷰 포인트:
  - rag_docs에서 삭제된 Markdown 파일의 chunk가 검색에 남는 문제를 해결한다.
  - source_paths가 빈 목록이면 rag_documents 전체를 비운다. rag_docs에 Markdown 파일이 하나도 없을 때는 의도한 동작이다.

함수: search_similar_chunks(embedding, top_k, distance_threshold=None) -> list[dict]
- 역할:
  - 질문 embedding과 rag_documents.embedding의 cosine distance를 계산해 가까운 chunk를 반환한다.
- 입력:
  - embedding: 질문 embedding.
  - top_k: 반환할 chunk 개수.
  - distance_threshold: 이 값보다 distance가 큰 chunk는 제외한다. None이면 threshold 없이 검색한다.
- 출력:
  - source_path, chunk_index, content, distance가 담긴 dict 목록.
- 리뷰 포인트:
  - distance threshold가 추가되어 관련성이 낮은 chunk는 prompt에 들어가지 않는다.
  - 기본 threshold는 retriever.py의 RAG_DISTANCE_THRESHOLD 값으로 관리한다.
  - threshold가 너무 낮으면 필요한 문서도 빠지고, 너무 높으면 무관한 문서가 들어갈 수 있다.
  - MariaDB Vector 인덱스가 실제로 사용되는지는 실행 계획 확인이 필요하다.

6. app/db/chat_memory.py
================================================================================

파일 역할
- MariaDB 기반 대화 메모리를 관리한다.
- 최근 대화 원문 유지, 오래된 대화 요약, 메시지 저장을 담당한다.

상수: SUMMARY_TRIGGER_TURNS = 6
- 의미:
  - 요약을 시작할 기준 턴 수.
  - unsummarized 메시지가 6턴을 초과하면 요약한다.

상수: RECENT_TURNS = 3
- 의미:
  - prompt에 원문 그대로 넣을 최근 대화 턴 수.

함수: create_session(user_id="local_user") -> int
- 역할:
  - chat_sessions에 새 세션을 만든다.
- 입력:
  - user_id
- 출력:
  - 새 session id.
- 리뷰 포인트:
  - user_id 형식 검증이 없다.
  - 현재는 로컬 단일 사용자라 큰 문제는 아니다.

함수: get_or_create_session(session_id, user_id="local_user") -> dict
- 역할:
  - session_id가 있고 DB에 있으면 기존 세션을 반환한다.
  - 없으면 새 세션을 생성한다.
- 입력:
  - session_id
  - user_id
- 출력:
  - id, user_id, summary, message_count가 있는 dict.
- 리뷰 포인트:
  - session_id가 존재하면 user_id 일치 여부를 확인하지 않는다.
  - 존재하지 않는 session_id가 오면 조용히 새 세션을 만든다. 사용자가 잘못된 session_id를 보냈는지 구분할 수 없다.

함수: add_message(session_id, role, content) -> None
- 역할:
  - chat_messages에 메시지를 저장하고 chat_sessions.message_count를 1 증가시킨다.
- 입력:
  - session_id
  - role: user 또는 assistant
  - content
- 출력:
  - 없음.
- 리뷰 포인트:
  - role 값 검증은 DB ENUM에 의존한다.
  - 사용자 메시지 저장과 assistant 메시지 저장이 각각 별도 트랜잭션으로 실행된다. 둘 중 하나만 저장될 가능성을 완전히 막지는 못한다.

함수: get_recent_turns(session_id, limit=RECENT_TURNS) -> list[dict[str, str]]
- 역할:
  - 아직 요약되지 않은 최근 메시지에서 user/assistant 쌍을 만들어 반환한다.
- 입력:
  - session_id
  - limit
- 출력:
  - [{"user": "...", "assistant": "..."}] 형식의 최근 대화 목록.
- 리뷰 포인트:
  - 최근 limit*2개 메시지만 가져와 쌍을 만든다.
  - user/assistant 순서가 깨진 데이터가 있으면 일부 메시지는 무시될 수 있다.
  - 현재 정상 API 흐름에서는 user와 assistant가 번갈아 저장되므로 MVP에서는 충분하다.

함수: get_unsummarized_turn_count(session_id) -> int
- 역할:
  - 아직 요약되지 않은 메시지 수를 세고 2로 나눠 턴 수로 계산한다.
- 입력:
  - session_id
- 출력:
  - 요약되지 않은 턴 수.
- 리뷰 포인트:
  - 메시지 수가 홀수인 경우 반올림 없이 버린다.
  - assistant 저장 실패 등으로 메시지 쌍이 깨지면 실제 턴 수와 다를 수 있다.

함수: get_turns_to_summarize(session_id, keep_recent_turns=RECENT_TURNS) -> list[dict]
- 역할:
  - 아직 요약되지 않은 메시지 중 최근 3턴을 제외한 오래된 메시지를 반환한다.
- 입력:
  - session_id
  - keep_recent_turns
- 출력:
  - 요약 대상 메시지 row 목록.
- 리뷰 포인트:
  - 메시지 개수 기준으로 자르기 때문에 user/assistant 쌍이 깨진 경우 요약 경계도 깨질 수 있다.

함수: update_summary(session_id, summary, summarized_message_ids) -> None
- 역할:
  - chat_sessions.summary를 갱신하고, 요약에 반영된 메시지들의 is_summarized를 TRUE로 바꾼다.
- 입력:
  - session_id
  - summary
  - summarized_message_ids
- 출력:
  - 없음.
- 리뷰 포인트:
  - IN 절은 placeholder를 사용해 SQL injection 위험은 낮다.
  - summary 업데이트와 메시지 플래그 업데이트가 같은 db_cursor 안에서 실행되어 트랜잭션으로 묶인다.

함수: maybe_summarize_session(session_id, summary, summarizer) -> str | None
- 역할:
  - 요약되지 않은 턴 수가 기준을 넘으면 summarizer를 호출해 summary를 갱신한다.
- 입력:
  - session_id
  - summary

8. app/rag/ingest_docs.py
================================================================================

파일 역할
- rag_docs 폴더의 Markdown 파일을 읽어 chunk로 나눈 뒤 embedding을 생성하고 MariaDB에 저장한다.

상수: PROJECT_ROOT
- 의미:
  - 프로젝트 루트 경로.

상수: RAG_DOCS_DIR
- 의미:
  - RAG 문서 폴더. 기본값은 프로젝트 루트의 rag_docs.

상수: CHUNK_SIZE
- 의미:
  - chunk 최대 문자 수. 기본값 900.

상수: CHUNK_OVERLAP
- 의미:
  - chunk 간 겹치는 문자 수. 기본값 150.

함수: chunk_text(text, chunk_size=CHUNK_SIZE, overlap=CHUNK_OVERLAP) -> list[str]
- 역할:
  - 긴 문서를 일정한 문자 수 단위로 자르고 overlap을 둔다.
- 입력:
  - text
  - chunk_size
  - overlap
- 출력:
  - chunk 문자열 목록.
- 리뷰 포인트:
  - 문장/제목/표 단위가 아니라 문자 수 기준으로 자른다.
  - Markdown 표나 항목이 중간에 잘릴 수 있다.
  - overlap이 chunk_size보다 크거나 같은 경우에도 무한 루프는 피하지만 chunk 품질은 나빠질 수 있다.

함수: ingest_markdown_file(path: Path) -> int
- 역할:
  - Markdown 파일 하나를 읽고 chunk별 embedding을 저장한다.
- 입력:
  - path: Markdown 파일 경로.
- 출력:
  - 저장한 chunk 수.
- 리뷰 포인트:
  - 저장 전에 delete_document_chunks(relative_path)를 호출해 기존 chunk를 먼저 삭제한다.
  - 따라서 파일이 수정되어 chunk 수가 줄어도 이전 chunk가 남지 않는다.
  - embedding 생성 중간에 실패하면 일부 chunk만 저장될 수 있다.

함수: ingest_all() -> None
- 역할:
  - rag_docs 아래 모든 .md 파일을 찾아 ingest_markdown_file을 실행한다.
- 입력:
  - 없음.
- 출력:
  - 없음. 진행 상황을 print로 출력한다.
- 리뷰 포인트:
  - 시작 시 delete_chunks_except_sources(source_paths)를 호출해 rag_docs에서 삭제된 문서의 DB chunk를 지운다.
  - rag_docs에 Markdown 파일이 하나도 없으면 rag_documents 전체를 비운다.
  - 대량 문서 처리 시 진행률, 실패 재시도, 로그 파일이 없다.

실행 블록: if __name__ == "__main__"
- 역할:
  - python -m app.rag.ingest_docs 실행 시 ingest_all()을 호출한다.

9. app/rag/retriever.py

파일 역할
- 사용자 질문을 embedding하고 MariaDB Vector 검색을 실행하는 RAG 검색 래퍼다.

상수: TOP_K

  - 무관한 질문에서는 빈 chunk 목록이 반환될 수 있고, 이 경우 prompt_builder.py는 "참고할 수 있는 정보가 없습니다"를 넣는다.
  - embedding 생성 실패와 DB 검색 실패를 구분해서 처리하지 않는다.

10. app/rag/prompt_builder.py
================================================================================

파일 역할
- 검색된 RAG chunk와 사용자 질문을 하나의 LLM prompt로 만든다.
- 답변 말투, 언어, 수치 생성 금지, 가격 답변 방식 등의 정책을 포함한다.

함수: build_prompt(question: str, chunks: list[dict]) -> str
- 역할:
  - RAG 참고 정보와 사용자 질문을 합쳐 LLM에 전달할 최종 user prompt를 만든다.
- 입력:
  - question: 사용자 질문.
  - chunks: retrieve_context가 반환한 chunk 목록.
- 출력:
  - prompt 문자열.
- 주요 정책:
  - 반드시 한국어로 답변.
  - 중국어, 한자, 긴 영어 설명 금지.
  - 사용자가 요청하지 않으면 "문서에 따르면", "RAG 문서" 같은 표현 금지.
  - 참고 정보에 없는 구체 수치나 기준값 생성 금지.
  - 가격 질문은 kg당 가격을 우선 답변하고 매입가/판매가를 구분.
- 리뷰 포인트:
  - chunk가 많아질수록 prompt가 길어진다.
  - prompt 안의 정책이 늘어나면 모델이 일부 지시를 놓칠 수 있다.
  - RAG context가 없을 때도 "참고할 수 있는 정보가 없습니다"를 넣고 답변 생성을 시도한다.

11. app/llm/ollama_client.py
================================================================================

파일 역할
- Ollama Qwen 모델 호출을 담당한다.
- 답변 생성, 한국어 외 텍스트 감지, 재번역, 대화 요약 기능을 제공한다.

상수: OLLAMA_BASE_URL
- 의미:
  - Ollama 서버 주소. 기본값 http://localhost:11434.

상수: CHAT_MODEL
- 의미:
  - 답변 생성에 사용할 Ollama 모델. 기본값 qwen2.5:7b.

정규식: HANJA_OR_CHINESE_PATTERN
- 역할:
  - CJK 한자 영역 문자를 감지한다.

정규식: ENGLISH_WORD_PATTERN
- 역할:
  - 2글자 이상 영어 단어를 감지한다.

정규식: HYPHENATED_ENGLISH_PATTERN
- 역할:
  - 하이픈이 들어간 영어 표현을 감지한다.

상수: ALLOWED_SHORT_ENGLISH_WORDS
- 역할:
  - AI, API, kg, RAG 같은 짧은 기술/단위 용어는 허용한다.

함수: contains_hanja_or_chinese(text: str) -> bool
- 역할:
  - 텍스트에 한자/중국어 문자가 포함되어 있는지 확인한다.
- 입력:
  - text
- 출력:
  - True 또는 False.
- 리뷰 포인트:
  - 한국어 문서에서 드물게 쓰는 한자도 중국어처럼 감지한다.

함수: contains_excessive_english(text: str) -> bool
- 역할:
  - 허용된 짧은 용어를 제외하고 영어가 너무 많이 포함되었는지 판단한다.
- 입력:
  - text
- 출력:
  - True 또는 False.
- 판단 기준:
  - 영어 단어가 5개 이상.
  - 영어 글자 수가 25자 이상이고 전체 비율이 25% 이상.
  - 하이픈 영어 표현 포함.
- 리뷰 포인트:
  - 영문 품종명, 업체명, 모델명 등은 과하게 차단될 수 있다.
  - 반대로 짧은 영어가 여러 번 반복되는 경우는 허용될 수 있다.

함수: contains_disallowed_foreign_text(text: str) -> bool
- 역할:
  - 한자/중국어 또는 과한 영어가 있는지 통합 판단한다.
- 입력:
  - text
- 출력:
  - True 또는 False.

함수: build_messages(prompt, history=None, summary=None) -> list[dict[str, str]]
- 역할:
  - Ollama chat API에 전달할 messages 배열을 만든다.
- 입력:
  - prompt: 현재 질문용 RAG prompt.
  - history: 최근 3턴 원문 대화.
  - summary: 오래된 대화 요약.
- 출력:
  - [{"role": "...", "content": "..."}] 목록.
- 구성:
  1. 시스템 역할/말투/언어 정책.
  2. 이전 대화 요약.
  3. 최근 user/assistant 대화.
  4. 현재 RAG prompt.
- 리뷰 포인트:
  - summary에 잘못된 오래된 정보가 들어가면 후속 답변에 영향을 줄 수 있다.
  - 최근 대화와 현재 RAG 정보가 충돌할 때 시스템 프롬프트는 현재 정보를 우선하라고 지시한다.

함수: generate_answer(prompt, history=None, summary=None) -> str
- 역할:
  - Qwen 모델에 messages를 보내 답변을 생성한다.
  - 답변에 중국어/한자/과한 영어가 있으면 재번역 요청을 한 번 더 보낸다.
- 입력:
  - prompt
  - history
  - summary
- 출력:
  - 최종 답변 문자열.
- 리뷰 포인트:
  - 답변 생성이 최대 2번의 LLM 호출이 될 수 있어 느려질 수 있다.
  - 재번역도 실패하면 고정 안내문을 반환한다.
  - Ollama 예외를 내부에서 처리하지 않고 상위로 올린다.

함수: summarize_conversation(existing_summary: str, rows: list[dict]) -> str
- 역할:
  - 기존 summary와 오래된 메시지 row들을 합쳐 1,200자 이내의 새 한국어 요약을 만든다.
- 입력:
  - existing_summary
  - rows: chat_messages row 목록.
- 출력:
  - 갱신된 summary 문자열.
- 리뷰 포인트:
  - summary 길이를 프롬프트로만 제한한다. 코드에서 강제로 자르지는 않는다.
  - 요약 모델이 숫자나 맥락을 잘못 압축하면 이후 후속 질문 품질이 떨어질 수 있다.
  - 요약 실패가 현재 /chat API 실패로 이어질 수 있다.

================================================================================
우선 리뷰하면 좋은 체크리스트
================================================================================

1. /chat에서 요약 실패가 답변 실패로 이어지는가?
   - app/api/chat.py의 maybe_summarize_session 호출을 분리할지 검토.

2. RAG 문서를 수정/삭제했을 때 오래된 chunk가 남는가?
   - 개선 완료: app/rag/ingest_docs.py와 app/db/vector_search.py에 삭제 로직이 추가되었다.
   - 남은 검토: embedding 생성 중간 실패 시 일부 chunk만 저장될 수 있는 문제.

3. embedding 모델과 VECTOR 차원이 항상 맞는가?
   - schema.sql의 VECTOR(1024)와 OLLAMA_EMBEDDING_MODEL 관계 확인.

4. 관련 없는 질문에도 chunk가 억지로 들어가는가?
   - 개선 완료: RAG_DISTANCE_THRESHOLD 기본값 0.52가 추가되었다.
   - 남은 검토: 실제 데이터가 늘어나면 threshold 재튜닝 필요.

5. session_id를 통한 다른 세션 접근 가능성이 있는가?
   - 다중 사용자 전환 전 user_id 검증 필요.

6. UI에서 내부 chunk를 계속 보여줄 것인가?
   - 개발자 디버그용과 사용자용 화면 분리 검토.

7. README가 실제 API와 실행 방식에 맞는가?
   - session_id, chat memory, 로컬 MariaDB 실행 절차 추가 검토.
